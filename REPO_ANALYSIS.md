# REPO_ANALYSIS.md

**项目名称：** VLM-data-agent（基于 Hermes-data-agent 演进）  
**阶段：** 第1阶段——分析与范围界定  
**日期：** 2026-04-14  
**作者：** Manus AI

## 一、任务背景与本阶段目标

当前仓库 `Hermes-data-agent` 已经具备较成熟的文本数据合成骨架，其核心链路为 **Seed → Generalization → Cascade Verify → Provenance → Dataset Export**。结合此前对 VisionFoundry 的方法与源码分析，可以确认：将其并入 Hermes-data-agent，演进为支持图像类数据泛化、图像问答样本生成与视觉一致性验证的 **VLM-data-agent**，在工程上是可行的，且最合理的路线不是“外挂一个视觉脚本”，而是将视觉能力重构进现有分层架构中。

本阶段不直接进入编码实现，而是完成三个前置目标。第一，明确当前仓库的可复用资产、边界与改造切入点。第二，定义 VLM-data-agent 的第一阶段功能范围，避免一开始把多图叙事、编辑修复、领域定制等高复杂度能力全部压入首个迭代。第三，按**技术依赖顺序**而不是表面重要性，输出后续开发的功能优先级、用户故事与机器可验证验收标准，作为第2阶段架构设计的输入。

## 二、当前仓库现状摘要

从现有 README、架构文档与核心模块实现可以看出，`Hermes-data-agent` 当前的主能力仍然聚焦于车载座舱话术数据合成，但它已经具备较强的框架化基础，因此适合作为多模态扩展的承载层。

| 维度 | 当前现状 | 对 VLM 演进的意义 |
| --- | --- | --- |
| 核心处理链路 | 已具备 SeedEngine、GeneralizationEngine、CascadeOrchestrator、ProvenanceTracker、OrbitDatasetAdapter | 视觉链路可以沿相同职责分层扩展 |
| 工具与技能机制 | 已有 `tools/` 注册模式与 `skills/` 技能定义 | 适合后续接入 `vlm_*` 工具和新技能 |
| CLI 入口 | `scripts/cli.py` 已支持多模式与 `orbit` 子命令 | 可以继续扩展 `vlm` 子命令而无需新建独立入口 |
| 验证思想 | 已具备“规则 → 语义 → 安全”的级联短路逻辑 | 可演进为多模态验证链 |
| 数据追踪 | 已支持来源链与验证链记录 | 适合纳入图像提示词、图像路径、VLM 判定轨迹 |
| 当前限制 | 数据模型、工具 schema 与 LLM 客户端仍以文本为中心 | 是首阶段最需要优先改造的部分 |

当前仓库的优点在于“骨架稳定”，而不是“视觉能力已具备”。因此，第1阶段要避免把目标表述成“给现有文本链路加几个图像字段”，而应明确这是一轮**数据契约升级 + 多模态客户端扩展 + 单图最小闭环落地**的系统性演进。

## 三、现有测试资产基线

当前仓库已经具备一定测试基础，但这些测试主要覆盖 ORBIT 文本链路与工具层行为，尚未形成面向视觉数据生成的测试体系。已确认的测试资产如下。

| 类别 | 现状 | 说明 |
| --- | --- | --- |
| 单元/模块测试 | `tests/test_orbit_core.py`、`tests/test_orbit_llm_mock.py` | 覆盖 ORBIT 核心逻辑与 LLM mock 场景 |
| 工具测试 | `tests/test_tools.py` | 覆盖既有工具层行为 |
| README 声明测试基线 | ORBIT 相关 76 个测试，核心覆盖率约 94% | 这是已有文本链路的质量基础，可作为后续回归测试基线 |
| 测试文档 | README 中有测试命令说明 | 说明仓库已有测试执行规范，但 VLM 尚无独立测试说明 |
| 多模态测试 | 暂无 | 第1阶段后续必须补齐 mock 化视觉链路测试 |

这意味着后续开发不应从“零测试”起步，但也不能误判为“现有测试足以覆盖 VLM 改造”。更准确的判断是：**文本 ORBIT 有测试基线，视觉 VLM 尚无测试基线**。

## 四、问题定义：VLM-data-agent 第1阶段到底要做什么

结合前期可行性分析，本项目的最终方向不是简单地在 Hermes-data-agent 中增加图像输入输出，而是使其具备**面向视觉语言模型训练数据的系统化生成能力**。不过在第1阶段，范围必须收敛到“单图模式最小可运行闭环”，否则项目会过早陷入多图叙事、编辑修复、场景模板、领域知识库等高复杂度细节。

因此，第1阶段的目标应定义为：

> 在不破坏现有文本 ORBIT 能力的前提下，为仓库增加一条**单图 VLM 数据合成闭环**，能够从视觉任务配置出发，生成图像问答样本，完成视觉一致性验证，并导出训练所需的基础注释格式。

这一目标意味着第1阶段更关注“系统骨架是否能承载视觉链路”，而不是“样本是否已经覆盖全部业务场景”。

## 五、第1阶段功能点与优先级

### 5.1 优先级原则

本项目采用 **技术依赖排序**，而不是简单的高、中、低优先级排序。也就是说，越靠前的功能不一定是用户最直观看见的功能，但一定是后续所有功能成立的前提。对 VLM-data-agent 来说，正确的依赖顺序应当是：

**数据契约/配置层 → 多模态客户端层 → 核心生成与验证层 → 导出层 → CLI/工具/技能接入层 → 高级能力扩展层。**

### 5.2 功能点优先级表

| 依赖顺序 | 优先级 | 功能点 | 目标说明 | 为什么必须先做 |
| --- | --- | --- | --- | --- |
| 1 | P0 | 多模态数据契约升级 | 定义 `VLMSample`、视觉任务配置、结构化输出载荷 | 没有统一数据模型，后续客户端、验证器和导出器都无法稳定协作 |
| 2 | P0 | 多模态客户端层 | 新增 `ImageClient` 与 `VLMClient`，抽象文本模型、图像模型与视觉验证模型 | 若仍沿用文本 `LLMClient`，视觉能力将难以维护且 provider 耦合过高 |
| 3 | P0 | 单图样本生成引擎 | 新增视觉种子/泛化逻辑，输出 question、answer、prompt、statement 等 | 这是 VLM-data-agent 的最小业务主链路 |
| 4 | P0 | 视觉验证与级联接入 | 对生成样本执行结构校验、QA/prompt 自洽校验、图像一致性验证 | 没有验证闭环，无法保证生成数据可用于训练 |
| 5 | P0 | 单图训练导出 | 导出基础 `annotations.json` 或兼容 `messages + images` 的训练格式 | 没有导出，闭环就无法真正落地到训练数据 |
| 6 | P1 | CLI / Tool / Skill 接入 | 新增 `vlm` 子命令、工具封装、技能入口 | 这是 Agent 化使用面的关键，但应建立在主链路跑通之后 |
| 7 | P1 | 追踪与审阅输出增强 | 扩展 provenance、增加审阅导出格式（如 CSV/Excel） | 便于人工质检与故障追溯，但不是主链路成立前的阻塞项 |
| 8 | P2 | 多图叙事与编辑修复 | 支持多图模式、失败后编辑修复、跨图一致性验证 | 复杂度高，不适合作为第1阶段承诺 |
| 9 | P2 | 车载视觉领域专有模板 | 引入舱内实体池、视角模板、业务规则库 | 有业务价值，但应在通用单图链路稳定后建设 |

从这个排序可以看出，第1阶段真正需要承诺的是 **P0 全量 + P1 的接口预留**，而不是一次性交付全部 P1/P2 功能。换言之，本阶段的交付标准不是“功能很多”，而是“主链路闭环清晰、架构延展方向正确”。

## 六、建议的第1阶段范围边界

为了保证第2阶段架构设计与第4阶段实现可控，第1阶段需要明确哪些能力**纳入当前迭代**，哪些能力**暂不纳入**。

| 类别 | 纳入本阶段 | 暂不纳入本阶段 |
| --- | --- | --- |
| 数据对象 | `VLMSample`、视觉任务配置、视觉验证结果结构 | 多图 story-chain 复杂对象、编辑历史链 |
| 模型能力 | 单图 prompt 生成、单图图像生成、单图视觉一致性校验 | 多图联合推理、编辑修复循环 |
| 导出能力 | 单图训练注释导出、基础审阅字段导出 | 全量多格式审阅报表模板 |
| 接入方式 | 预留 `vlm` 子命令与工具层接入点 | 完整 Agent 技能生态自动编排 |
| 场景覆盖 | 通用视觉感知/视觉问答任务 | 车载 DMS/OMS 专项对象体系 |

这个边界的意义在于：如果第1阶段就把多图、编辑修复、领域模板全部纳入，那么项目会在数据模型与验证逻辑上迅速膨胀，反而拖慢主链路上线时间。

## 七、建议的用户故事（按技术依赖顺序）

以下用户故事用于指导后续阶段设计与实现。每个故事都被控制在可以由单次实现任务独立完成的粒度内，并附带机器可验证的验收条件。

### US-01：定义多模态样本与配置契约

作为开发者，我希望在仓库中引入统一的视觉样本与任务配置数据结构，以便后续模块可以围绕同一份结构化契约协作。

| 项目 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 主要产物 | `VLMSample`、视觉任务配置 schema、验证结果 schema |
| 验收标准 | 能以结构化对象表示 `question`、`answer`、`image_prompt`、`image_path(s)`、`statement`、`metadata`；类型检查通过；相关单元测试通过；不破坏现有文本 ORBIT 测试 |

### US-02：新增多模态客户端抽象

作为开发者，我希望把文本生成、图像生成和视觉验证调用拆分为清晰的客户端接口，以便后续可切换不同 provider 而不污染核心业务逻辑。

| 项目 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 主要产物 | `ImageClient`、`VLMClient`、配置映射接口 |
| 验收标准 | 可通过 mock provider 完成单图生成与视觉判定调用；失败重试与异常结构统一；静态分析通过；客户端层自动化测试通过 |

### US-03：实现单图视觉种子与泛化能力

作为数据生成操作者，我希望系统可以从视觉任务配置生成单图样本草稿，包括问题、答案、图像提示词与语义陈述，以便形成可验证的候选样本。

| 项目 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 主要产物 | `visual_seed_engine` 或扩展后的 `seed_engine`，`visual_generalization_engine` |
| 验收标准 | 给定最小配置后可稳定产出结构完整的候选样本；候选样本字段不缺失；自动化测试通过；对现有文本流水线无回归 |

### US-04：实现视觉验证闭环并接入级联编排

作为质量控制者，我希望系统能够对候选视觉样本执行多阶段验证，以便过滤掉不自洽或与图像内容不一致的数据。

| 项目 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 主要产物 | `visual_verifier`、级联编排适配逻辑 |
| 验收标准 | 至少具备结构校验、问答/提示词自洽校验、图像一致性校验三类结果；验证失败样本可被明确标记或拦截；自动化测试通过 |

### US-05：导出单图训练数据格式

作为训练工程师，我希望系统能把通过验证的样本导出为基础训练注释格式，以便直接用于后续多模态 SFT 或数据审阅。

| 项目 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 主要产物 | 单图 `annotations.json` 或等价训练格式导出器 |
| 验收标准 | 导出文件中包含图像路径与问答消息；格式可被独立脚本加载验证；自动化测试通过 |

### US-06：为 CLI 和工具层预留 VLM 入口

作为操作者，我希望在现有仓库中通过统一入口触发 VLM 数据合成流程，以便项目继续保持 Hermes 风格的使用体验。

| 项目 | 内容 |
| --- | --- |
| 优先级 | P1 |
| 主要产物 | `vlm` CLI 子命令占位、配置样例、工具层接口草案 |
| 验收标准 | CLI 能识别新模式或预留参数；配置加载不报错；自动化测试通过 |

### US-07：补齐视觉链路最小测试基线

作为维护者，我希望为新增视觉链路建立 mock 化自动化测试，以便后续重构不会破坏最小闭环。

| 项目 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 主要产物 | 视觉样本 schema 测试、客户端 mock 测试、导出测试、流程 smoke test |
| 验收标准 | 新增测试可在无真实外部 API 条件下运行；所有新增测试通过；原有 ORBIT 测试继续通过 |

以下测试用例面向“用户能否使用第1阶段成果完成最小闭环”的角度设计，将贯穿后续实现、反思与测试阶段。

## 九、第2阶段架构设计输入

基于上述分析，第2阶段架构设计应重点回答以下问题，而不是重新讨论“要不要做 VLM”。

| 架构问题 | 第2阶段必须回答的内容 |
| --- | --- |
| 数据契约 | `VLMSample` 与现有 `OrbitRecord` 的关系如何定义 |
| 模型调用层 | `LLMClient`、`ImageClient`、`VLMClient` 如何分层与配置 |
| 编排层 | 现有 `CascadeOrchestrator` 如何兼容视觉验证阶段 |
| 导出层 | 单图训练导出与审阅导出是否拆分为双轨输出 |
| 兼容性 | 如何保证文本 ORBIT 与 VLM 能力共存 |
| 测试层 | 如何让视觉链路在无真实 API 的情况下完成大部分回归测试 |

## 十、本阶段结论

本阶段结论可以概括为三点。第一，`Hermes-data-agent` 作为承载层是合适的，因为它已经具备清晰的分层骨架、验证编排思想和数据追踪机制。第二，`VLM-data-agent` 的首个实现目标必须收敛为**单图最小闭环**，而不是一开始追求多图叙事和编辑修复。第三，后续开发优先级必须遵循 **数据契约 → 客户端 → 生成/验证 → 导出 → 接入层** 的依赖顺序，否则实现会被耦合和返工拖慢。

因此，建议进入第2阶段架构设计，并围绕本文件中的 P0 功能点与用户故事开展系统设计。
