# Car-ORBIT-Agent — 产品规格说明

## 产品愿景

Car-ORBIT-Agent 是 Hermes-data-agent 平台中的一个新 Agent 模块，借鉴 ORBIT 框架（Open-Web Reasoning for Information Retrieval Tasks）的核心方法论——**系统化种子驱动、多维度变体生成、级联多层验证、完整可追溯性**——并将其改造为适用于车载座舱语音助手的 NLU 训练数据合成流水线。该模块旨在从车辆功能树和参数库出发，系统化地生成覆盖长尾场景的高质量话术变体，并通过规则验证、语义验证和安全验证三层级联机制确保数据质量，最终输出带有完整来源链和验证链的可追溯训练数据集。

## 目标用户

本模块的主要用户是**座舱 AI 语音交互团队的数据工程师和 NLU 算法工程师**。他们需要大规模、高覆盖、高质量的话术变体数据来训练和评估车载语音助手的意图识别和槽位填充模型。次要用户是**产品经理和测试工程师**，他们需要通过可追溯的数据来源链来审核数据质量和覆盖度。

## 核心功能（MVP）

### 功能 1：种子生成（Seed Generation）

- **描述**：从车辆功能树配置文件（YAML/JSON）和参数库中，通过组合策略系统化地生成功能种子列表。每个种子包含领域、功能、参数组合和标准话术，作为后续泛化的输入。支持从现有 Excel 数据中提取种子，也支持从功能树配置自动组合生成。
- **用户故事**：作为数据工程师，我希望从车辆功能树配置文件自动生成覆盖所有功能和参数组合的种子列表，以便系统化地覆盖长尾场景而不遗漏。
- **验收标准**：
  1. 当提供有效的功能树 YAML 配置文件时，执行种子生成，应输出包含领域、功能、参数组合、标准话术的种子列表（JSON 格式）。
  2. 当提供现有的 Excel 数据文件时，执行种子提取，应正确解析每行数据为种子对象。
  3. 当配置文件格式错误或缺少必要字段时，应抛出明确的错误信息并终止。
- **优先级**：高

### 功能 2：多维度泛化（Multi-dimensional Generalization）

- **描述**：对每个种子，调用 LLM 沿多个维度（口语化程度、句式变化、参数变化、简化省略、驾驶场景适配）生成话术变体。每个维度可独立配置是否启用和生成数量。该功能复用现有的 LLM 调用基础设施（OpenAI 兼容 API）。
- **用户故事**：作为 NLU 算法工程师，我希望为每条标准话术自动生成覆盖多个表达维度的变体，以便训练数据能够反映真实用户的多样化表达方式。
- **验收标准**：
  1. 当提供一个有效种子和默认维度配置时，执行泛化，应为该种子生成指定数量的变体，且变体覆盖至少 3 个不同维度。
  2. 当指定仅启用部分维度时，生成的变体应只包含已启用维度的变化。
  3. 当 LLM API 调用失败时，应触发重试机制（最多 3 次），最终失败则记录日志并返回空列表。
- **优先级**：高

### 功能 3：级联验证（Cascade Verification）

- **描述**：对生成的变体执行三层级联验证流水线：（1）规则验证——基于纯 Python 逻辑检查参数完整性、长度约束、车辆约束规则；（2）语义验证——调用 LLM 判断变体与原始话术的语义一致性和自然度；（3）安全验证——调用 LLM 检查变体是否存在驾驶安全风险。每层验证独立评分，最终汇总为综合置信度分数。
- **用户故事**：作为数据工程师，我希望生成的每条变体都经过多层自动化验证，以便确保最终数据集的质量满足生产标准而无需大量人工审核。
- **验收标准**：
  1. 当提供一组变体和对应的种子信息时，执行级联验证，应依次执行规则验证、语义验证、安全验证，并输出每层的通过/失败状态和原因。
  2. 当变体缺少关键参数时，规则验证应将其标记为失败，且该变体不进入后续语义验证阶段。
  3. 当所有验证通过时，应输出综合置信度分数（0.0-1.0）。
- **优先级**：高

### 功能 4：来源链与验证链追踪（Provenance & Verification Chain）

- **描述**：为每条最终输出的变体记录完整的来源链（来源种子、领域、功能、参数组合、生成策略）和验证链（每层验证的结果、分数、原因），输出为结构化 JSON 格式。
- **用户故事**：作为产品经理，我希望每条生成的训练数据都有完整的来源和验证记录，以便在数据审核时能够追溯问题并持续改进生成流程。
- **验收标准**：
  1. 当完整流水线执行完成时，每条输出记录应包含 `source_chain` 和 `verification_chain` 字段。
  2. 当某条变体在验证中被拒绝时，其验证链应记录拒绝的阶段和具体原因。
- **优先级**：高

### 功能 5：CLI 命令与批量处理

- **描述**：在现有 `hermes-data` CLI 中新增 `orbit` 子命令，支持单条处理和批量处理模式。批量模式复用现有的 `BatchRunner` 基础设施，支持并行处理、断点续传和轨迹保存。
- **用户故事**：作为数据工程师，我希望通过命令行一键启动车载 ORBIT 流水线的批量处理，以便高效地生成大规模训练数据集。
- **验收标准**：
  1. 当执行 `hermes-data orbit --input data/seeds.yaml --output output/orbit/` 时，应完成完整的种子生成→泛化→验证流水线并输出结果。
  2. 当使用 `--limit N` 参数时，应只处理前 N 个种子。
  3. 当使用 `--resume` 参数时，应从上次中断的位置继续处理。
  4. 当处理完成时，应在终端输出统计摘要（总种子数、总变体数、各层验证通过率）。
- **优先级**：高

### 功能 6：Hermes 技能与工具集注册

- **描述**：将 car-orbit-agent 的能力封装为 Hermes Agent 生态的标准技能（SKILL.md）和工具集（Toolset），使其可以被 Hermes Agent 的 AIAgent 循环和 BatchRunner 直接调用。
- **用户故事**：作为平台开发者，我希望 car-orbit-agent 的功能以标准的 Hermes 技能和工具集形式注册，以便与现有的 Agent 编排和自进化机制无缝集成。
- **验收标准**：
  1. 当 Hermes Agent 加载工具集时，应能发现并注册 `car-orbit` 工具集中的所有工具。
  2. 当 Agent 读取 `car-orbit-synthesis` SKILL.md 时，应获得完整的流水线执行指南。
- **优先级**：中

## 非功能范围

本模块明确不做以下事情：

- **不实现 ORBIT 原始的问答对生成流程**：本模块不生成搜索代理的问答对，而是专注于车载话术变体。
- **不实现网络爬虫或 Selenium 自动化**：不复刻 ORBIT 原论文中的 DeepSeek Chat 网页自动化方式。
- **不实现 NLU 模型训练**：本模块只负责生成训练数据，不包含模型训练逻辑。
- **不实现实时车辆状态查询**：安全验证基于静态规则和 LLM 判断，不连接真实车辆 API。
- **不替换现有的 cockpit-utterance-synthesis 功能**：car-orbit-agent 是一个独立的、增强的流水线，与现有功能并行存在。

## 技术栈

| 组件 | 技术选型 | 选型理由 |
|------|----------|----------|
| 编程语言 | Python 3.11 | 与现有项目一致 |
| LLM 接口 | OpenAI 兼容 API（openai 库） | 复用现有基础设施，支持 gpt-4.1-mini/nano 等模型 |
| 数据处理 | pandas + openpyxl | 复用现有依赖，支持 Excel 和 DataFrame 操作 |
| 配置管理 | PyYAML | 复用现有依赖，用于功能树和参数库配置 |
| CLI 框架 | Click + Rich | 复用现有 CLI 基础设施 |
| 测试框架 | pytest + pytest-asyncio | 复用现有测试基础设施 |
| Agent 生态 | hermes-agent（可选依赖） | 复用 BatchRunner、delegate_task、ToolRegistry |

## 关键假设

1. 车辆功能树和参数库由用户以 YAML 或 JSON 格式提供，或从现有 Excel 数据中提取。
2. LLM API（OpenAI 兼容）可用且稳定，模型支持中文生成和理解。
3. 现有的 Hermes-data-agent 架构（技能系统、工具注册、BatchRunner）保持稳定，不会在开发期间发生破坏性变更。
4. 安全验证规则基于静态规则库和 LLM 判断，不需要连接真实车辆系统。

## 已知风险

| 风险 | 影响程度 | 发生可能性 | 缓解策略 |
|------|----------|------------|----------|
| LLM 生成的变体质量不稳定 | 高 | 中 | 通过三层级联验证过滤低质量数据；支持配置不同模型用于生成和验证 |
| 功能树配置的组合爆炸 | 中 | 中 | 提供 `max_seeds` 参数限制种子数量；支持采样策略而非全量组合 |
| hermes-agent 依赖不可用 | 低 | 中 | 所有核心功能（种子生成、泛化、验证）不依赖 hermes-agent；Agent 集成作为可选增强 |
| API 调用成本过高 | 中 | 低 | 默认使用 gpt-4.1-nano 进行规则验证；支持配置不同模型用于不同验证阶段 |

## 成功标准

1. 种子生成能够从功能树配置中系统化地覆盖所有领域和功能组合。
2. 多维度泛化生成的变体在口语化、句式、参数变化等维度上具有明显的多样性。
3. 级联验证能够有效过滤掉参数缺失、语义偏移和安全风险的变体，最终数据集的质量评分（人工抽检）达到 90% 以上。
4. 完整的来源链和验证链使每条数据可追溯。
5. CLI 命令支持批量处理，处理 1000 条种子的端到端时间在合理范围内（取决于 API 速率）。
6. 所有核心功能的单元测试覆盖率达到 80% 以上。
